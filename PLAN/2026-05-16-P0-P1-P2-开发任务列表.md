# 当前简历面试 Agent P0 / P1 / P2 开发任务列表

> 来源：基于《当前简历面试 Agent 待完善与优化清单》进一步拆分出的可执行任务清单。
> 
> 目标：把“问题描述”变成可以进入开发排期、逐项验收的任务列表。

---

## 使用说明

- `P0`：优先保证链路稳定、协议清晰、能持续跑通。
- `P1`：优先提升面试质量、题目命中率、评分与复盘质量。
- `P2`：优先提升产品完成度、长期可维护性与分析能力。
- 每个任务都尽量写成“可直接实现”的粒度，便于继续拆成 issue 或 sprint task。

### 任务拆分补充规则

- 以下“细化任务拆分”默认以非测试代码为口径控制粒度；测试补齐跟随对应实现切片一起完成，但不单独作为拆分依据。
- 单个子任务的目标范围是 100 到 200 行左右的非测试代码改动；如果预估超过 300 行，则继续向下拆分。
- 单个子任务尽量只影响 1 个 owner 文件加少量直接依赖文件，避免一次横跨前端、BFF、Mastra 三层。
- 需要跨层联动的功能，优先拆成“契约定义”“单层接入”“状态透传”“展示或落盘”四类子任务，分别落地。
- 已完成任务保持原有完成状态不变；本次新增的细化拆分主要用于后续复盘、补票和继续演进。

## P0 开发任务列表

## P0-1 启动协议结构化改造（已完成）

**目标**

- 去掉当前依赖 kickoff message 拼接文本的初始化协议。
- 让前端、BFF、Mastra 之间通过结构化请求对象通信。

**为什么要做**

- 当前初始化严重依赖拼接文本和字段文案稳定性，属于脆弱协议，任何一层小改都可能把整条启动链路弄断。
- 现在的契约分散在前端类型、BFF 拼接逻辑和 Mastra 文本解析里，排障和演进成本都偏高。

**做完的好处**

- 启动链路会从“靠约定字符串”变成“靠明确 schema”，后续改字段、改文案、改启动流程都更安全。
- 前端、BFF、Mastra 的边界会更清晰，测试、调试、重构和联调都会轻很多。

**具体任务**

- [x] 定义统一的 interview start request schema，覆盖 `threadId`、`resumeMarkdown`、`jobDescriptionMarkdown`、`settings`、`resumeSections` 等字段。
- [x] 在前端统一使用该 schema 生成启动请求，不再默认依赖自然语言格式拼接。
- [x] 在 BFF 中取消初始化文案协议拼接逻辑，改为结构化透传或轻量映射。
- [x] 在 Mastra 侧为 `initialize-session` 新增结构化输入分支。
- [x] 保留旧协议的兼容读取一段时间，避免一次性切断现有链路。
- [x] 为新旧协议切换增加日志标记，便于观察真实流量已全部迁移。

**细化任务拆分（供后续复盘 / 复用）**

- 前端先抽出统一的 interview start request 类型与构造函数，只解决字段命名和默认值，不改动流式链路。
- 前端发送启动请求时切到结构化 payload，仅替换 kickoff 拼接入口，避免同时改 UI 状态管理。
- BFF 先补齐启动请求 schema 与解析逻辑，让控制器和 service 共享同一个输入契约。
- BFF 再移除旧的初始化文案拼接逻辑，改成结构化透传或轻量映射，避免与 schema 变更同批次落地。
- Mastra 单独增加 `initialize-session` 的结构化输入分支，先保证新协议可走通，再处理旧协议兼容读取。
- 新旧协议切换日志与兼容兜底单独作为一个切片收尾，避免把观测代码和主链路改动混在一起。

**建议改动范围**

- `frontend/src/types/agent.ts`
- `frontend/src/services/agent-stream.ts`
- `frontend/src/views/AgentChatView.vue`
- `bff/src/modules/agent/agent.schemas.ts`
- `bff/src/modules/agent/agent.service.ts`
- `src/mastra/tools/interview-state-manager-tool.ts`

**验收标准**

- 新增启动请求可以不依赖字符串字段名解析直接完成初始化。
- 删除或修改 kickoff 文案后，启动链路仍能稳定工作。
- 前端、BFF、Mastra 三层 schema 不再出现重复定义且互相漂移。

## P0-2 统一简历解析入口（已完成）

**目标**

- 消除 `resume-parser-tool` 与 state manager 内部解析逻辑分叉。
- 建立单一权威的简历解析模块。

**为什么要做**

- 现在存在多套解析逻辑并行演进，规则一旦分叉，就会出现 BFF 认为合法、Mastra 却理解成另一套结构的风险。
- 重复实现意味着任何解析优化都要多处同步，维护成本高且容易遗漏。

**做完的好处**

- 简历结构提取结果会更一致，后续选题、统计和诊断都能建立在同一份标准化数据上。
- 解析规则以后只需要维护一处，迭代速度更快，出错面更小。

**具体任务**

- 盘点当前所有简历解析入口，标明谁在主链路上真正生效。
- 把专业技能、项目经历、技能组计数、标题兼容等解析逻辑收敛到一个模块。
- 让 state manager 初始化只能消费标准化解析结果，而不是再次从 raw markdown 里切字符串。
- 明确 parser 输出模型，例如 `professionalSkillsSection`、`projectExperienceSection`、`normalizedSkills`、`warnings`。
- 删除或降级重复实现，保留一个 canonical parser。
- 为 parser 增加非标准 markdown 的回归测试样本。

**细化任务拆分（供后续复盘 / 复用）**

- 先盘点并标记简历解析入口，输出主链路 owner，再决定哪些实现保留、哪些转成适配层。
- 单独定义 canonical parser 输出模型，先稳定 `professionalSkillsSection`、`projectExperienceSection`、`normalizedSkills`、`warnings` 等字段。
- 将标题识别与 section 切分逻辑收敛到 canonical parser，不和技能归一化一起改。
- 将技能提取、技能组计数和项目经历提取分别迁入 canonical parser，按能力块逐个迁移。
- BFF 上传校验先改为消费 canonical parser 输出，不同时处理 Mastra 初始化逻辑。
- state manager 初始化再切到标准化解析结果输入，移除对 raw markdown 的二次字符串解析。
- 最后把旧实现降级成 wrapper 或删除，确保仓内只剩一个权威解析入口。

**建议改动范围**

- `src/mastra/tools/resume-parser-tool.ts`
- `src/mastra/tools/interview-state-manager-tool.ts`
- `bff/src/modules/resume/resume-markdown.ts`
- 共享解析逻辑的新模块文件

**验收标准**

- 项目中只有一个真正负责简历结构提取的权威实现。
- 修改解析规则时，不需要同步改两处以上核心逻辑。
- 同一份简历在 BFF 校验和 Mastra 初始化中的解析结果一致。

**当前完成情况**

- 当前单一权威实现已收敛到 `bff/src/modules/resume/resume-parser.ts`。
- BFF 上传校验、技能组计数、启动 payload `resumeSections` 预填、Mastra kickoff recovery、`resume-parser-tool` 与 state manager 初始化都已复用同一 parser 输出。
- parser 已覆盖标题别名、不同 heading style、编号列表、嵌套列表、无列表标记逐行回退等非标准 markdown 场景，并已有回归测试样本。

## P0-3 问题生成主链路架构收敛（Planner-Retriever-Generator + 状态机 + Critic/Judge）

**目标**

- 把当前“状态机内混合规划、召回、组题、判定”的问题生成流程，收敛为可演进的生产级主链路：Planner 负责规划，Retriever 负责召回，Generator 负责生成，Critic/Judge 负责校验与裁决，状态机只负责编排和推进。

**为什么要做**

- 现在问题生成职责较多地堆在核心状态机和相邻逻辑里，边界不清会让 JD 接入、覆盖控制、追问策略、评分校验互相耦合。
- 如果不先把问题生成主链路拆成稳定 owner，后续再做覆盖规划、难度梯度、评分 rubric、trace 优化时，很容易继续在一个大流程里叠分支。
- 生产环境里真正决定题目质量的，不只是“能不能召回到题”，而是“先规划什么、按什么取、如何组题、由谁拦截低质量题”，这些环节都需要显式化。

**做完的好处**

- 问题生成链路会从“隐式混合逻辑”变成“阶段清晰、可观测、可替换”的架构，后续扩展 JD、难度控制、rerank、专项模式都会更稳。
- 每道题都可以追踪来源：规划意图、召回依据、生成结果、裁决结论，排障和优化成本会明显下降。
- 状态机可以回到“流程真相来源”的单一职责，不再同时承担过多选题细节。

**具体任务**

- [x] 明确 Planner 输出契约，至少包含本轮目标能力、题型、覆盖意图、JD/简历信号、预期难度和选择原因。
- [x] 将 Retriever 从状态推进逻辑中拆出，统一负责 query 构造、metadata 过滤、召回结果归一和 trace 记录。
- [x] 将 Generator 明确为“基于规划结果 + 召回候选 + 当前状态”生成最终问题与追问文案的独立阶段。
- [x] 引入 Critic/Judge 阶段，对生成题做最小质量闸门，例如相关性、重复度、覆盖冲突、与当前轮次或难度是否匹配；不通过时触发重试或 fallback。
- [x] 让状态机只编排阶段切换、持久化上下文、处理异常恢复和非法状态转移，不再直接承载规划细节与生成细节。
- [x] 为 Planner、Retriever、Generator、Critic/Judge 分别增加单测，并补一条主链路集成测试覆盖“规划 -> 召回 -> 生成 -> 裁决 -> 状态推进”。

**细化任务拆分（供后续复盘 / 复用）**

- 先抽 Planner 输出契约与类型定义，只落地 `targetAbility`、`questionType`、`coverageIntent`、`selectionReason` 等结构，不改召回逻辑。
- 把简历信号和 JD 信号组装收敛到 Planner 输入适配层，避免状态机直接拼 planner context。
- Retriever 第一刀只抽出 query builder，让 query 构造从状态推进函数中脱离。
- Retriever 第二刀再补 metadata filter、候选归一化和 recall trace 记录，不和 query builder 同批次扩容。
- Generator 第一刀只负责主问题生成输入装配与结果结构化，不同时处理追问文案。
- Generator 第二刀再承接 follow-up 生成，把回答分析到追问 prompt 的映射独立出去。
- Critic/Judge 先落最小质量闸门和 fallback 规则，再补 richer 的裁决理由和 trace 字段。
- 状态机最后只保留阶段编排、上下文持久化、异常恢复和非法转移控制，删除内联规划与生成细节。

**当前完成情况**

- Planner 已输出结构化规划契约，当前每个 plan 已显式携带 `targetAbility`、`questionType`、`coverageIntent`、`resumeSignals`、`jobDescriptionSignals`、`expectedDifficulty` 与 `selectionReason`，不再只有 skill / lens 这种隐式信息。
- 初始化阶段的 Retriever 已从 state manager 中抽到独立模块，当前由 retriever 统一负责 query 构造、召回结果聚合和 recall trace 收集，state manager 只消费结构化 retrieval result。
- Generator 已独立负责两类生成：初始化主问题集合的最终适配，以及基于当前状态与回答分析生成追问文案；state manager 不再内联维护追问 prompt 与模型调用细节。
- Critic/Judge 已在初始化主问题生成后生效，当前会拦截空题、重复题、明显目标错位或轮次形状不匹配的问题，并稳定替换为确定性的 fallback 题目，避免低质量题直接进入状态机。
- 初始化主链路已被抽成独立 pipeline，当前 state manager 只负责调用 pipeline、恢复/持久化 working memory、更新 outcome 与 rag sample，并继续承接后续状态推进与异常恢复。
- 当前已为 Planner、Retriever、Generator、Critic/Judge 分别补齐单测，并新增一条“规划 -> 召回 -> 生成 -> 裁决 -> 状态推进”的集成测试；针对本次重构切片执行的 coverage 已达到 95.7% branch / 98.87% lines。

**建议改动范围**

- `src/mastra/tools/interview-state-manager-tool.ts`
- `src/mastra/lib/interview-question-planner.ts`
- `src/mastra/lib` 下新增或拆分 question planner / retriever / generator / judge 相关模块
- `src/mastra/tools/interview-question-tool.ts` 或相邻问题生成入口
- 相关单元测试与状态机脚本测试

**验收标准**

- 主链路中至少能清晰区分 Planner、Retriever、Generator、Critic/Judge、状态机五个职责边界。
- 任意一道生成出来的问题，都能追溯其规划输入、召回候选、最终生成结果与裁决结论。
- 当 Critic/Judge 判定题目不合格时，系统能够稳定 fallback，而不是把低质量题直接发给用户。
- 后续接入 JD、覆盖规划、难度调节和评分 rubric 时，不需要继续把复杂逻辑堆回状态机核心文件。

## P0-4 提升简历格式容错能力

**目标**

- 让“根据简历面试”不再强依赖严格模板化 markdown。

**为什么要做**

- 真实用户的简历格式往往不标准，如果系统过分依赖模板，就会把大量本来可用的简历挡在门外。
- 格式脆弱会直接伤害产品可用性，用户会把“格式问题”理解成“系统不懂我的简历”。

**做完的好处**

- 可接受的简历范围会明显扩大，上传成功率和面试启动率都会提升。
- 系统会更像“理解简历内容”，而不是“检查模板格式”，用户信任感也会更好。

**具体任务**

- 补充对中文/英文/中英混合标题别名的识别。
- 兼容无空行、嵌套列表、列表符号不统一、标题层级不规范等格式。
- 把简单的结构错误转换成 `warnings`，而不是直接完全失败。
- 对无法解析但可读的内容建立降级策略，例如全文扫描关键词 + 近邻段落聚类。
- 新增一组真实世界样本简历用于回归验证。

**细化任务拆分（建议 issue 粒度）**

- 先补标题别名字典，单独处理中文、英文和中英混合标题映射，不动 section 解析主流程。
- 增加输入标准化步骤，统一空行、缩进、列表符号和 heading 样式，降低后续解析分支数。
- 单独补嵌套列表与无空行场景的 section tokenizer，让结构切分先更稳。
- 再处理列表符号不统一、标题层级不规范等弱格式输入，限制在解析器内部局部规则里。
- 将轻度结构错误从 hard fail 改为 `warnings` 输出，单独改错误分级与返回模型。
- 最后增加无法完整解析时的降级提取策略，例如全文扫描关键词和近邻段落聚类。

**建议改动范围**

- `bff/src/modules/resume/resume-markdown.ts`
- `src/mastra/tools/resume-parser-tool.ts`
- 相关测试文件

**验收标准**

- 至少支持数种非模板 markdown 简历通过主链路启动面试。
- 对轻度格式错误不再直接阻断整个流程。
- 解析失败时能返回明确错误原因或降级结果。

## P0-5 让 JD 真正进入选题与召回链路

**目标**

- 让 JD 从“透传上下文”升级为“参与问题规划的核心输入”。

**为什么要做**

- 现在 JD 虽然传到了后端，但并没有真正影响问题规划，这会让“岗位定制”停留在表面。
- 如果只围绕简历自述出题，系统很难验证候选人与目标岗位之间的真实匹配度。

**做完的好处**

- 问题会更贴近目标岗位，面试结果也更有业务价值，而不只是泛泛的技术问答。
- 系统能同时覆盖“你会什么”和“岗位真正要什么”，区分度会更强。

**具体任务**

- [x] 提取 JD 中的关键职责、技术要求、优先技能、领域词。
- [x] 在专业技能轮 query 构造中引入 JD 关键词加权。
- [x] 在项目经历轮引入“JD 要求 × 项目证据”的交叉验证逻辑。
- [x] 为缺口能力生成专项验证问题，而不是只围绕简历自述发问。
- [x] 在 outcome 中记录某题是由简历驱动、JD 驱动还是交叉驱动。

**细化任务拆分（供后续复盘 / 复用）**

- 先抽 JD 原文信号提取器，只产出职责、技术要求、优先技能和领域词。
- 再定义 JD 信号标准模型，区分已对齐能力、缺口能力和岗位权重词。
- 在专业技能轮先改 query builder，让 JD 关键词加权进入召回，但不立即改生成策略。
- 项目经历轮单独增加“JD 要求 × 项目证据”交叉验证上下文，避免与专业技能轮共用一批逻辑。
- 缺口能力专项验证题作为独立 Planner 分支落地，不与原有简历驱动题混写在一起。
- outcome 与 generation trace 的 `questionDriver`、信号来源和选择原因单独补齐，保证后续可追踪。

**建议改动范围**

- `src/mastra/tools/interview-state-manager-tool.ts`
- `src/mastra/tools/interview-question-tool.ts`
- 与 query planning 相关的 lib 文件
- `frontend/src/views/AgentChatView.vue` 的配置预览文案

**验收标准**

- 上传 JD 与不上传 JD，问题规划结果会产生可解释的差异。
- 至少部分题目明确来自 JD 与简历交叉验证，而非只来自简历。
- 结果文件中可追踪每道题的触发依据。

**当前完成情况**

## P0-7 会话恢复能力（已完成）

**目标**

- 支持浏览器刷新或中断后恢复最近一次面试会话。

**为什么要做**

- 模拟面试本身是长流程任务，只要刷新、断网、误关闭页面，就有很大概率让用户前功尽弃。
- 虽然后端已有 thread 和 outcome 数据，但前端没有利用这些能力，实际体验仍然很脆弱。

**做完的好处**

- 用户即使中断也能继续，挫败感会明显下降，整条链路的可用性会提升一个层级。
- 后端已有状态数据能够真正发挥价值，而不是只作为内部存档存在。

**具体任务**

- 将 `threadId`、最近配置、会话状态摘要持久化到本地存储。
- 页面进入时检测是否存在未完成的最近会话。
- 增加“恢复上次面试 / 放弃并重新开始”的 UI。
- 对已完成会话与进行中会话分别处理。
- 若后端无法恢复，给出明确错误信息并清理无效本地状态。

**细化任务拆分（建议 issue 粒度）**

- 先定义前端会话持久化 schema，只落地 `threadId`、最近配置、阶段摘要和更新时间。
- 抽一个本地存储 service，负责读写最近会话信息，不把序列化细节塞进 store。
- 在面试启动成功后写入最近会话摘要，先覆盖 happy path，不立即处理恢复 UI。
- 页面初始化时单独增加未完成会话检测逻辑，只产出“有无可恢复会话”的判断结果。
- 再接入“恢复上次面试 / 放弃并重新开始”的 UI 交互，避免把展示层和存储层混在一批改动里。
- 最后补后端恢复失败时的清理逻辑和错误提示，保证不会停留在半初始化状态。

**当前完成情况**

- 前端已新增本地会话持久化 schema 与 `interview-session-storage` service，当前会稳定保存最近一次面试的 `threadId`、系统设置、阶段摘要与更新时间，并对不可用或损坏的 `localStorage` 数据做 fail-safe 清理。
- Agent 对话页已在收到权威 `interviewState` 后写入最近会话摘要，因此刷新或浏览器中断后，前端不再只依赖内存态保留当前线程。
- 路由层当前已支持“存在可恢复本地会话”时继续进入 `/agent`，不再因为上传页 store 已丢失就直接把用户拦回上传页。
- Agent 页面已新增“恢复上次面试 / 放弃并重新开始”交互；对未完成会话会恢复 thread、最近设置、当前题目与侧边栏进度摘要，对已完成会话则只提示清理本地记录后重新开始。
- 当前恢复逻辑已把展示层与纯恢复逻辑拆开：会话摘要格式化、恢复快照重建和失效 thread 判定已收敛到独立 service，避免继续堆在页面组件里。
- 当恢复后的首次续答被后端返回 `thread not found` 或 `missing thread` 一类错误时，前端现在会清理无效本地状态，给出明确提示，并在需要时回退到上传页重新开始，避免停留在半初始化状态。

**建议改动范围**

- `frontend/src/views/AgentChatView.vue`
- `frontend/src/router/index.ts`
- `frontend/src/services/interview-session-storage.ts`
- `frontend/src/services/interview-session-recovery.ts`

**验收标准**

- 刷新后能恢复最近一次未完成面试的 thread 及基本上下文。
- 恢复失败时不会出现半初始化状态。
- 同一个 thread 不会被重复错误初始化。

## P1-1 技能组归一化与去重

**目标**

- 将“技能组数量”升级为“可用于出题的规范化能力主题集合”。

**为什么要做**

- 原始技能组数量本身是高噪声信号，简历里写得多不代表真的有那么多值得考察的主题。
- 如果不做归一化，后续题量规划、召回策略和覆盖控制都会建立在不稳定的输入上。

**做完的好处**

- 出题会更聚焦于真实能力主题，而不是被词面重复和堆砌技能干扰。
- 题量和选题质量会更稳定，也更容易解释为什么系统要考这些点。

**具体任务**

- 抽取简历中的技能候选项并做别名归一化。
- 区分核心技能、辅助工具、项目中偶然出现的技术词。
- 合并重复或强相关技能，例如框架名与语言名的层级关系。
- 输出规范化后的技能主题列表及置信度。
- 为出题规划暴露“推荐题量上限”和“技能优先级”。

**细化任务拆分（建议 issue 粒度）**

- 先从简历解析结果中抽取原始技能候选项，形成统一的 skill token 列表。
- 单独建立别名归一化映射，把同义词、缩写和大小写变体收敛到 canonical name。
- 再增加核心技能、辅助工具、项目偶发现技术词的分类逻辑，不和归一化映射同批处理。
- 框架与语言、平台与工具之间的层级合并规则作为独立切片落地，避免一次写过多启发式。
- 最后输出面向 Planner 的结构化技能主题、置信度和题量上限建议。

**验收标准**

- 技能堆砌简历不会直接导致出题数异常膨胀。
- 相似技能不会重复触发相似问题。
- 题目规划可消费结构化技能主题而不是原始字符串列表。

## P1-2 基于 Planner 的题目覆盖策略优化

**目标**

- 在已完成主链路架构收敛的前提下，把题目覆盖从“有规划”推进到“规划质量稳定且可解释”。

**为什么要做**

- P0 解决的是职责边界和主链路稳定性，但并不自动保证覆盖策略足够好。
- 如果 Planner 只有粗粒度规划，没有明确覆盖配额、优先级和反重复约束，面试仍然可能出现偏斜或遗漏。

**做完的好处**

- 面试会更均衡，能更稳定覆盖关键能力面，而不是只依赖召回偶然性或少量启发式规则。
- 覆盖目标、已覆盖进度和剩余缺口都能被解释，后续调优会更直接。

**具体任务**

- 为专业技能轮定义覆盖维度，例如基础原理、工程实践、排障、权衡、优化。
- 为项目经历轮定义覆盖维度，例如 ownership、决策、复杂度、结果、复盘。
- 在 Planner 中引入覆盖配额、优先级与 diversity 约束，避免连续命中同一主题或同一提问视角。
- 在状态中记录已覆盖、待覆盖和被跳过的维度，用于驱动后续问题规划。
- 为每个问题记录规划原因、覆盖目标和被选中原因，供后续 trace 与 outcome 使用。
- 增加离线样本验证，检查不同简历 / JD 组合下的覆盖完整度和重复率。

**细化任务拆分（建议 issue 粒度）**

- 先为专业技能轮定义覆盖维度枚举和元数据，不立即接 Planner。
- 项目经历轮的覆盖维度单独建模，避免两个轮次共用一套过度抽象的枚举。
- Planner 先接入覆盖配额与优先级，不同时引入 diversity 约束。
- diversity 约束另做一刀，专门控制连续命中同主题或同提问视角的问题。
- 状态层单独增加已覆盖、待覆盖、被跳过维度的 ledger，供后续规划读取。
- 每题的覆盖目标、规划原因和被选中原因最后再写入 trace 与 outcome。

**验收标准**

- 同一场面试的问题重复率明显下降。
- 关键维度不会因召回偶然性被完全漏掉。
- 可以解释每道题对应的覆盖目标，以及为什么当前轮次优先问它。

## P1-3 难度梯度与动态调节

**目标**

- 让面试过程具备节奏与层次，而不是平铺直叙。

**为什么要做**

- 没有难度梯度的面试容易变成随机问答，既不像真实面试，也无法根据候选人表现做有效区分。
- 面试体验如果没有节奏，用户会觉得系统要么太浅，要么一上来就不合理地过难。

**做完的好处**

- 面试过程会更像真人面试官的推进方式，先验证基础，再拉高难度或回退确认。
- 最终评分也会更有说服力，因为问题强度本身是可追踪、可解释的。

**具体任务**

- 为候选问题补充难度标签、能力层级标签。
- 定义每轮默认的难度推进顺序。
- 根据回答质量动态决定下一题是升难、持平还是回退。
- 避免一开始就连续抛高难问题或全程停留在表层验证。
- 在结果文件中记录实际难度轨迹。

**细化任务拆分（建议 issue 粒度）**

- 先给候选问题或规划结果补难度标签，不立即调整选题策略。
- 再补能力层级标签，让难度和能力维度分开建模。
- 为每一轮定义默认的难度推进顺序，先作为静态策略接入 Planner。
- 回答质量到“升难 / 持平 / 回退”的动态策略单独实现，不和标签建模同批次完成。
- 最后把实际难度轨迹落入 outcome，方便回溯策略是否生效。

**验收标准**

- 面试过程能体现明显的由浅入深或动态调整逻辑。
- 高质量回答与低质量回答会影响下一题选择。
- 难度轨迹可在 outcome 中追踪。

## P1-4 Judge 评分 rubric 工程化

**目标**

- 在已有 Critic/Judge 框架上，把答题评分规则做成稳定、结构化、可复核的 Judge 输出。

**为什么要做**

- P0 里的 Critic/Judge 只解决“有没有最低质量闸门”和“评价链路有无 owner”，并不等于评分规则已经足够稳定。
- 如果 Judge 没有明确 rubric 和结构化输出，评分仍然会过度依赖模型临场判断。

**做完的好处**

- 评分会更稳定、更可解释，也更容易和具体改进建议对齐。
- 后续能力分析、成长趋势、报告生成都能直接复用 Judge 的结构化结果。

**具体任务**

- 按题型建立评分 rubric，例如基础概念题、项目题、系统设计题。
- 将评分拆为准确性、完整性、深度、结构、工程意识、表达清晰度等子项。
- 明确 Critic 与 Judge 的职责边界：Critic 负责题目质量闸门，Judge 负责回答质量评分与总结。
- 为 Judge 输出定义结构化 schema，并在 outcome 中存储子项分数、聚合分数与评分依据。
- 明确不同追问下的评分规则，例如主问题与追问的分值关系。
- 为 Judge 输出增加结构校验、异常兜底和一致性测试。

**细化任务拆分（建议 issue 粒度）**

- 先按题型定义 rubric 模板，只稳定基础概念题、项目题、系统设计题三类输入输出。
- 再抽评分子项 schema，把准确性、完整性、深度、结构、工程意识、表达清晰度做成统一维度。
- Critic 和 Judge 的职责边界单独梳理为适配层，避免题目闸门与答案评分继续共用同一函数。
- Judge 结构化输出 schema 单独落地，先打通子项分数和聚合分数的落盘。
- 主问题与追问的分值关系另起一个切片处理，避免和基础 rubric 设计绑在一起。
- 结构校验、异常兜底和一致性约束最后收口，防止 schema 仍在变化时反复返工。

**验收标准**

- 同类问题的评分维度一致，不依赖模型临时发挥。
- 结果文件可以解释“为什么这题得这个分”。
- 后续分析可以直接基于 Judge 的分项分数聚合。

## P1-5 追问策略分型

**目标**

- 让 follow-up 更像真实面试官的验证式追问。

**为什么要做**

- 如果追问类型单一，系统很难区分“答得不全”和“其实没真正理解”这两类完全不同的问题。
- 真实面试的区分度往往就来自追问，而不是首问本身。

**做完的好处**

- 系统会更像真正的面试官，不只是机械补问，而是能验证深度、ownership 和边界理解。
- 对候选人能力的判断会更立体，尤其能更好识别模板化回答和表层理解。

**具体任务**

- 将追问区分为补充细节、验证原理、追查 ownership、边界挑战、方案比较、故障排查等类型。
- 针对项目题增加“你本人做了什么”的 ownership 追问模板。
- 针对疑似背题答案增加反模板化追问。
- 建立何时停止追问并切题的规则，避免过度纠缠。
- 在结果文件中记录追问类型和触发原因。

**细化任务拆分（建议 issue 粒度）**

- 先建立 follow-up 类型枚举与统一输出契约，只解决“追问属于哪一类”。
- 项目题的 ownership 追问模板单独落地，避免和其它追问类型一起调整 prompt。
- 反模板化追问单独接入触发规则，只处理疑似背题或套话场景。
- 停止追问并切题的策略独立实现，防止追问生成与追问终止耦合在一起。
- 追问类型与触发原因最后写入 outcome 和 trace，保证后续可分析。

**验收标准**

- 追问不再只有“你没答全，再补充一下”这一类模式。
- 项目题能有效区分真实参与和表层描述。
- 追问记录可以回溯和分析。

## P1-6 低质量输入与语音噪声分类增强

**目标**

- 更精细处理短回答、噪声文本、空话和可挽回的跑题。

**为什么要做**

- 在真实场景里，用户输入并不总是干净完整的，尤其语音输入会带来大量噪声和碎片文本。
- 如果系统把所有低质量输入都按同一种方式处理，就很容易出现误判和不公平评分。

**做完的好处**

- 系统会对真实使用场景更鲁棒，不会因为一句噪声或一句短答就把整题判偏。
- 用户会感受到系统是在帮助自己继续答好，而不是因为输入不完美就直接扣分。

**具体任务**

- 为极短回答、ASR 噪声、口语碎片、模板空话建立单独分类。
- 对可挽回的跑题优先发澄清式追问，而不是直接低分结束。
- 区分“内容正确但表达差”和“表达顺但内容空”的评分路径。
- 为语音输入场景增加专门清洗步骤。

**细化任务拆分（建议 issue 粒度）**

- 先建立短回答、ASR 噪声、口语碎片、模板空话的输入分类器，不立即改变评分逻辑。
- 语音输入清洗单独抽成预处理步骤，先处理重复词、停顿词和明显转写噪声。
- 对可挽回跑题的澄清式追问策略单独接入，避免与输入分类器共改同一批条件分支。
- “内容正确但表达差”和“表达顺但内容空”的区分另做一层评分输入特征，不把所有判断都堆进 Judge prompt。
- 阈值、分类标签和透传字段最后统一收口，便于后续调参。

**验收标准**

- 明显噪声文本不会直接污染评分链路。
- 可挽回回答有机会被引导回正轨。
- 同类低质量输入的处理策略更稳定。

## P1-7 Planner / Retriever / Generator trace 闭环优化

**目标**

- 让 P0 已产出的规划、召回、生成、裁决 trace 真正服务于问题质量迭代。

**为什么要做**

- P0 解决的是“先把 trace 留下来且链路可追踪”，但如果这些数据不进入分析闭环，就仍然只是日志。
- 没有把规划意图、召回候选、生成结果、裁决结论和最终面试表现挂起来，就很难判断问题到底是“规划错了”“召回偏了”还是“生成质量不够”。

**做完的好处**

- 题目优化会从经验驱动变成数据驱动，后续调 Planner 权重、Retriever filter、Generator prompt 或 Judge 闸门都有依据。
- 团队可以更快定位质量问题发生在哪个阶段，而不是把所有问题都归到“模型不稳定”。

**具体任务**

- 建立 Planner 意图、Retriever 候选、Generator 最终题目、Critic/Judge 结论与最终答题表现之间的映射关系。
- 标记“规划命中但召回弱”“召回充分但生成偏”“题目相关但区分度低”“重复问题”“偏题问题”等标签。
- 输出各阶段调优所需的聚合统计，例如不同 planner lens、不同 query 模板、不同 filter 命中效果。
- 回灌 Planner 权重、metadata 过滤、rerank 策略、Generator prompt 和 Judge 闸门调整建议。

**细化任务拆分（建议 issue 粒度）**

- 先统一 Planner、Retriever、Generator、Critic/Judge 的 trace id 与题目关联键。
- 再建立 trace 到最终答题表现的映射表，让单题能够串起各阶段数据。
- 阶段质量标签单独定义，如“规划命中但召回弱”“召回充分但生成偏”，避免直接散落在分析脚本里。
- 聚合统计输出单独实现，分别面向 planner lens、query 模板、filter 命中效果生成统计结果。
- 回灌调参建议作为最后一层派生逻辑，不和原始 trace 聚合写在同一个模块。

**验收标准**

- trace 不再只是日志留存，而是可分析的数据资产。
- 能通过数据区分是 Planner、Retriever、Generator 还是 Judge 环节表现差。
- 后续问题质量优化有客观依据而不是拍脑袋。

## P1-8 最终报告与 outcome 导出增强

**目标**

- 提升单场面试的复盘价值。

**为什么要做**

- 当前 outcome 数据已经很丰富，但原始结构并不适合直接给用户或面试教练阅读。
- 如果单场结果不能快速复盘，系统的训练价值会大打折扣。

**做完的好处**

- 用户可以更直接看到自己每道题的亮点、短板和后续练习方向，复盘成本会明显下降。
- 同一份结果既能给人看，也能给后续分析或推荐逻辑复用，信息密度更高。

**具体任务**

- 生成按题目维度展开的结果摘要。
- 生成按能力维度聚合的强弱项总结。
- 增加“下次优先补强主题”和推荐练习项。
- 支持 Markdown / HTML 导出结果。
- 增加“完整对话 + 结构化评分”的统一导出入口。

**细化任务拆分（建议 issue 粒度）**

- 先生成按题目维度展开的结果摘要，只解决单题评分和点评的人类可读输出。
- 能力维度聚合总结另做一刀，避免把题目摘要和能力聚合混在同一个 formatter 中。
- “下次优先补强主题”和推荐练习项作为独立推荐模块生成，不和摘要模板直接耦合。
- Markdown 导出先落地，优先形成稳定的文本报告格式。
- HTML 导出在 Markdown 产物稳定后再接，避免双模版同时调整。
- “完整对话 + 结构化评分”的统一导出入口最后补齐，作为汇总层而不是底层 formatter。

**验收标准**

- 用户可直接阅读结果文件完成复盘。
- 不需要再从原始 JSON 手工提取关键信息。
- 报告同时支持人读和后续机器分析。

## P1-9 feedback schema 治理与迁移

**目标**

- 收敛 outcome 中反馈字段的演进痕迹。

**为什么要做**

- outcome schema 一旦出现多个历史字段并存，后续任何读写逻辑都会变复杂，而且容易出现兼容 bug。
- 现在不治理，之后每增加一轮需求都会继续堆技术债。

**做完的好处**

- outcome 结构会更稳定，后续导出、聚合、分析和迁移都会简单很多。
- 团队可以更放心地演进结果模型，而不是每次都担心旧文件被读坏。

**具体任务**

- 统一 feedback 字段归属与命名。
- 清理 `candidateImprovement.feedback` 与 `userFeedback` 并存的历史兼容结构。
- 定义 schemaVersion 迁移策略。
- 为旧 outcome 文件提供迁移或兼容读取逻辑。
- 增加 outcome 读写一致性测试。

**细化任务拆分（建议 issue 粒度）**

- 先定义 canonical feedback 字段与命名，明确新写入只认一个权威位置。
- 再改写 outcome writer，让新增结果文件统一写入 canonical feedback 字段。
- 旧字段兼容读取单独做 reader adapter，避免读写逻辑同步大改。
- `schemaVersion` 迁移策略与迁移 helper 独立落地，不和 reader adapter 混写。
- 历史 outcome 的回填或兼容清理最后处理，保证线上已有文件仍可稳定读取。

**验收标准**

- 新旧 outcome 文件都能被稳定读取。
- feedback 不再存在多处重复语义字段。
- 后续 schema 调整有清晰迁移路径。

## P2 开发任务列表

## P2-1 流式异常恢复与幂等重试

**目标**

- 提升 SSE 链路在弱网和中断场景下的可恢复性。

**为什么要做**

- 流式对话是当前主链路的核心，一旦 SSE 脆弱，用户会直接感知为“系统老是断”。
- 没有恢复和幂等策略时，重复点击和重试还会带来状态错乱，比单纯失败更危险。

**做完的好处**

- 弱网和偶发中断不再必然导致整轮作废，产品稳定性感知会显著改善。
- 前端重试行为会更安全，不容易出现重复推进状态机的问题。

**具体任务**

- 识别前端中断类型：网络错误、代理断开、上游 4xx/5xx、解析错误。
- 在 UI 中增加“重试本轮”“重新连接”“重新开始本轮”的明确操作。
- 基于 `threadId` 和请求语义设计幂等重试策略。
- 避免重复发送回答导致双写或状态推进两次。
- 为流式失败场景补充集成测试。

**细化任务拆分（建议 issue 粒度）**

- 先建立 SSE 前端错误分类，把网络错误、代理断开、上游 4xx/5xx、解析错误区分开。
- UI 的“重试本轮”“重新连接”“重新开始本轮”按钮单独落地，只依赖错误分类结果。
- 幂等重试策略单独定义请求语义和 `threadId` 绑定规则，不先改按钮行为。
- 发送层再接入请求去重与重复提交保护，避免双写或状态推进两次。
- 流式恢复与幂等策略稳定后，再为失败场景补集成测试与日志断言。

**验收标准**

- 流中断后用户有明确恢复路径。
- 同一轮不会因为重复点击造成状态错乱。
- SSE 失败类型可区分显示。

## P2-2 语音输入增强

**目标**

- 提升语音输入在真实面试使用场景下的稳定性。

**为什么要做**

- 语音输入是很自然的面试交互方式，但浏览器原生能力在专业术语、断句和错误恢复上都偏弱。
- 如果语音能力不好，用户会把问题归因到“系统评价不准”，因为输入内容本身就已经被污染了。

**做完的好处**

- 语音作答会更接近真实面试体验，用户不必总是切回键盘。
- 转写质量提升后，后续追问、评分和总结也都会更准确。

**具体任务**

- 增加专业术语词表与转写纠错。
- 增加“转写确认后发送”的模式。
- 对口语停顿、重复词、错误断句做清洗。
- 区分语音输入失败、权限失败、浏览器不支持三类状态。
- 评估是否引入更稳定的语音服务而非只依赖浏览器原生能力。

**细化任务拆分（建议 issue 粒度）**

- 先增加专业术语词表与基础纠错映射，只解决常见技术词识别。
- 口语停顿、重复词、错误断句的清洗单独落到 transcript preprocessor。
- “转写确认后发送”作为独立交互模式接入，不和底层识别逻辑同批次修改。
- 语音输入失败、权限失败、浏览器不支持三类状态单独建模并映射到 UI。
- 如果需要切换更稳定的语音服务，先抽 provider adapter 边界，再评估替换实现。

**验收标准**

- 专业术语识别准确率明显改善。
- 用户可以在发送前修正转写结果。
- 语音输入错误不会直接污染评分。

## P2-3 面试快捷操作与可解释性 UI

**目标**

- 提升用户在面试前、中、后的操作效率和理解成本。

**为什么要做**

- 现在很多动作都只能靠用户自己想办法输入或切换页面，操作门槛偏高。
- 当前状态虽然能从数据里推出来，但对用户并不够透明，导致流程掌控感偏弱。

**做完的好处**

- 用户可以更轻松地驱动流程、查看上下文、导出结果和重新开始，使用门槛会明显下降。
- 面试过程会变得更可解释，用户知道系统现在在考什么、还剩什么、可以做什么操作。

**具体任务**

- 增加“查看识别到的技能组”“导出面试记录”“复制最终报告”“重新开始但保留简历”等操作。
- 增加“当前考察目标”“本轮剩余题数”“当前题所属主题”的更强提示。
- 增加“我不理解问题”“请换个说法”“结束当前轮次”等快捷意图按钮。
- 在开始前展示结构化面试预览摘要。

**细化任务拆分（建议 issue 粒度）**

- 先增加面试中的操作栏骨架，只预留查看技能组、导出、复制报告、重新开始等动作入口。
- “查看识别到的技能组”做成独立面板或弹层，不与导出功能同批开发。
- “导出面试记录”“复制最终报告”“重新开始但保留简历”分别按独立命令处理，避免一个 action handler 过大。
- 当前考察目标、本轮剩余题数、当前题所属主题作为单独的上下文提示条接入。
- “我不理解问题”“请换个说法”“结束当前轮次”等快捷意图按钮单独映射到已有输入协议。
- 开始前的结构化面试预览摘要最后再补，作为预览层汇总已有状态。

**验收标准**

- 用户无需全靠自由输入也能控制流程。
- 面试的当前状态和目标更加透明。
- 复盘和重新开始操作更顺手。

## P2-4 outcome 聚合分析层

**目标**

- 从单场结果文件升级到跨场次分析能力。

**为什么要做**

- 单场面试只能回答“这次怎么样”，无法回答“长期哪里一直薄弱、哪里在进步”。
- 如果没有聚合层，系统就很难从练习工具升级为真正的成长型训练系统。

**做完的好处**

- 系统能输出长期趋势和反复出现的薄弱项，训练价值会大幅提升。
- 后续个性化推荐、阶段性复盘和能力画像都会有稳定的数据底座。

**具体任务**

- 建立 interview history 聚合索引。
- 汇总每次面试的总分、强项、弱项、技能主题表现、用户反馈。
- 支持按时间、技能、轮次、题型做趋势分析。
- 输出候选人的持续薄弱主题与成长轨迹。

**细化任务拆分（建议 issue 粒度）**

- 先定义 interview history 聚合索引结构，只解决单场 outcome 如何被纳入历史索引。
- 再补基础聚合字段，如总分、强项、弱项、技能主题表现和用户反馈。
- 按时间维度的趋势分析单独实现，避免和主题聚合一起拉高复杂度。
- 按技能、轮次、题型的筛选与查询能力分别拆开落地，不做一个大而全的分析接口。
- 持续薄弱主题与成长轨迹作为基于聚合结果的派生分析，最后单独输出。

**验收标准**

- 系统能够回答“过去几次面试哪些方面持续薄弱”。
- 可以按主题或时间查看趋势，而不是只能看单次 JSON。
- 聚合层不破坏现有单场 outcome 设计。

## P2-5 能力维度可视化与复盘面板

**目标**

- 让结果从数据文件变成产品级反馈视图。

**为什么要做**

- 即使 outcome 数据足够完整，如果展示层不直观，用户也很难快速理解自己的能力结构。
- 纯 JSON 或纯长文报告不适合做长期训练反馈，信息吸收成本太高。

**做完的好处**

- 用户可以更快看懂自己在哪些维度强、哪些维度弱，复盘和决策效率更高。
- 结果页会更像完整产品输出，而不是后台记录的可视化附属品。

**具体任务**

- 定义能力维度模型，例如基础知识、项目表达、系统思维、排障、追问承压。
- 将单题表现映射到能力维度。
- 生成能力雷达图或分维度趋势图。
- 设计简洁的面试后复盘面板或报告页面。

**细化任务拆分（建议 issue 粒度）**

- 先定义能力维度模型，明确基础知识、项目表达、系统思维、排障、追问承压等维度。
- 单题表现到能力维度的映射单独实现，先形成稳定的 dimension score view model。
- 雷达图和分维度趋势图拆成两个独立展示任务，避免一个图表组件承接过多职责。
- 面试后复盘面板作为独立页面或面板接入，只消费已经稳定的图表数据和摘要数据。

**验收标准**

- 用户可以直观看到自己在哪些维度更强或更弱。
- 能力维度映射有据可查，不是随意展示。
- 图形化结果和原始 outcome 数据一致。

## P2-6 状态机编排层模块化落地

**目标**

- 在 P0 已明确“状态机只负责编排”的前提下，把 `interview-state-manager-tool` 的实现真正收口到编排层职责。

**为什么要做**

- 如果 P0 只在设计层面划清了边界，但实现里仍把规划、召回、生成、裁决、持久化混在一个大文件里，回归风险仍然很高。
- 大文件和高耦合会让后续任何局部改动都继续牵一发而动全身。

**做完的好处**

- 模块边界会和实际代码结构对齐，每一层都更容易单测、替换和重构。
- 新功能接入时不必每次都深入一个巨型核心文件，维护成本会明显下降。

**具体任务**

- 将状态迁移校验、阶段编排、上下文持久化、fallback 处理拆成独立模块。
- 把 Planner、Retriever、Generator、Critic/Judge 的调用适配层从核心状态机文件中抽离。
- 保留统一外部入口，但内部改为组合小型服务或纯函数。
- 为编排层与各阶段适配层补单元测试。
- 明确每一层输入输出模型，避免隐式共享状态。

**细化任务拆分（建议 issue 粒度）**

- 先抽状态迁移校验模块，只把合法状态转移判断从核心文件移出去。
- 阶段编排逻辑单独抽成 orchestration service，不同时迁移持久化逻辑。
- 上下文持久化适配层再单独拆分，明确读写 working memory、outcome、thread 状态的边界。
- fallback 处理独立成模块，避免与阶段编排共用一个超长控制函数。
- Planner、Retriever、Generator、Critic/Judge 的调用适配层逐个抽离，每次只迁一个 stage owner。
- 最后统一输入输出模型，消除隐式共享状态和跨模块直接读写。

**验收标准**

- 核心文件体积和职责显著下降。
- 单个功能变更不再需要读完整个大文件。
- 状态机实现与 P0 定义的职责边界保持一致。

## P2-7 依赖与 memory 兼容治理

**目标**

- 减少对底层依赖行为 bug 的长期本地绕过。

**为什么要做**

- 现在的 workaround 虽然有效，但本质上说明底层依赖行为并不完全受控，后续升级风险较高。
- 如果不把兼容边界讲清楚，团队很容易在升级时踩回同一个坑。

**做完的好处**

- 依赖升级会更有把握，不容易因为底层 memory 行为变化导致静默回归。
- workaround 会从“隐性知识”变成“明确治理对象”，维护风险更低。

**具体任务**

- 明确当前可工作的 Mastra / memory 版本组合。
- 记录 `ReadOnlyThreadMemory` 的存在原因和退役条件。
- 尝试向上游验证或修复 `readOnly` 配置问题。
- 补 memory 读写行为的回归测试。
- 在升级依赖前增加兼容性检查清单。

**细化任务拆分（建议 issue 粒度）**

- 先固化当前可工作的 Mastra / memory 版本矩阵，形成明确的兼容基线。
- `ReadOnlyThreadMemory` 的职责、存在原因和退役条件单独文档化并靠近实现位置。
- 将 workaround 封装在单独适配层中，避免业务代码继续直接依赖它。
- 上游 `readOnly` 配置问题的复现与验证脚本独立处理，不和业务修复混在一起。
- memory 读写回归测试与升级前兼容性检查清单最后一起补齐，形成升级护栏。

**验收标准**

- 团队清楚知道 workaround 为何存在、何时可以删除。
- 升级依赖时有可执行的回归保障。
- memory 行为不会在库升级后悄悄变化。

## P2-8 质量观测、健康检查与关联日志

**目标**

- 建立更清晰的排障和优化基础设施。

**为什么要做**

- 现在链路已经跨前端、BFF、Mastra 和本地持久化，多层系统没有统一观测就很难快速定位问题。
- 没有指标和关联日志，很多优化只能靠主观感觉，难以判断是否真的变好。

**做完的好处**

- 出问题时可以更快知道故障发生在哪一层，联调效率会大幅提升。
- 性能和质量优化会有客观指标支撑，后续迭代更可控。

**具体任务**

- 为 BFF 和 Mastra 增加 health / readiness endpoint。
- 为每次面试生成 correlation id，并贯穿前端、BFF、Mastra 日志。
- 汇总启动耗时、单题耗时、RAG 耗时、流完成率、错误率等指标。
- 设计最基础的调优面板或日志查询维度。

**细化任务拆分（建议 issue 粒度）**

- 先给 BFF 增加 health endpoint，只回答进程和基础依赖是否可用。
- Mastra readiness endpoint 单独实现，避免和 BFF 健康检查走同一个适配层。
- correlation id 的生成与透传单独打通，从前端请求到 BFF 再到 Mastra 日志逐层接入。
- 启动耗时、单题耗时、RAG 耗时、流完成率、错误率等指标分别抽成独立采集点，避免一个大而全的 metrics 模块。
- 调优面板或日志查询维度最后再设计，只消费已经稳定输出的结构化日志和指标。

**验收标准**

- 链路失败时可以快速定位卡在哪一层。
- 日志能够按 threadId 或 correlation id 聚合。
- 性能优化有基础指标可看。

## 一句话总结

如果要快速把当前系统从“能跑”提升到“可持续演进”，最应该先做的是四件事：先把启动协议和解析链路结构化，再把问题生成主链路收敛到 Planner / Retriever / Generator + 状态机 + Critic/Judge，然后让 JD 真正进入规划与召回，最后用全链路 E2E 把这条链路钉住。
